# dotbep Runtime

A **runtime** for dotBEP includes the effects, automations, and resolver handlers that give the BEP dynamic behavior on the dotbep platform, plus custom lens components and adapter functions for displaying remote data.

## How it works

A BEP (BIM Execution Plan) on dotbep has three layers:

1. **Structured data** — the `.bep` file (managed from the platform).
2. **Runtime** — this repository. Handlers that run on the dotbep server.
3. **Lenses** — Web Components that display remote data in the browser (also in this repo).

---

## Setup

```bash
cp .env.example .env
# Edit .env with DOTBEP_TOKEN and DOTBEP_BEP_ID
npm install
npm run pull      # downloads BEP types → src/bep.d.ts
```

`npm run pull` regenerates `.env.example` in place, adding a placeholder line for every env var this BEP's handlers declare (on top of the two fixed platform credentials). Re-run it and diff `.env.example` against your `.env` whenever a handler needs a new secret — see [Environment variables](#environment-variables).

## Scripts

All of these run through `@dotbep/cli` (the `dotbep` binary), installed as a devDependency — the scaffolded project's own `package.json` scripts are thin wrappers around it.

| Script | What it does |
|--------|-------------|
| `npm run pull`           | Downloads BEP types → `src/bep.d.ts`, built-in lens schemas → `src/lenses/built-in.ts`, version stamp → `meta.json`, and refreshes `.env.example` and `src/lenses/remote-data.json` — does *not* touch `DESIGN.md`, see [Custom lens design](#custom-lens-design) |
| `npm run dev`             | Starts a local preview harness (`dev/`) for testing custom lenses against real remote data, without deploying — run `npm run pull` first |
| `npm run build:runtime`  | Bundles `src/runtime/` → `dist/runtime.cjs` |
| `npm run build:lenses`   | Bundles `src/lenses/` → `dist/lenses.js` |
| `npm run deploy:runtime` | Builds then uploads the runtime to the BEP server |
| `npm run deploy:lenses`  | Builds then uploads the lenses bundle |

## Project structure

```
DESIGN.md                           ← OPTIONAL — only present if the runtime owner committed one to override the platform's design system, see "Custom lens design"
meta.json                           ← bepVersion this runtime was generated for — do not edit (regenerated by pull)
dev/                                ← local lens-preview harness (npm run dev) — do not need to edit
vite.dev.config.ts                  ← dev server config backing npm run dev
src/
├── bep.d.ts                        ← auto-generated types — do not edit
├── runtime/
│   ├── index.ts                    ← BepRuntime class; registers all handlers
│   ├── effects/                    ← one file per effect
│   ├── automations/                ← one file per automation
│   └── resolvers/                  ← one file per resolver
└── lenses/
    ├── index.ts                    ← barrel; export all lens components and adapters here
    ├── mappings.ts                 ← generated by pull; fill in lenses and adapters per remote data
    ├── remote-data.json            ← snapshot used by npm run dev — do not edit (regenerated by pull)
    ├── hello-lens.ts               ← example lens (safe to delete)
    ├── schemas.ts                  ← Zod schemas for lens data validation
    ├── built-in.ts                 ← built-in lens schemas — do not edit (regenerated by pull)
    └── adapters/                   ← one file per adapter (pure functions, run in browser)
```

The template ships with one working example of each handler type (`hello-*`). Delete them once you have implemented the real handlers for the BEP.

## Guide for the AI agent

### Your main task

Implement the handlers in `src/runtime/`, fill in the lens mappings in `src/lenses/mappings.ts`, and write custom lens components and adapters in `src/lenses/` if needed. All handler IDs the BEP requires are in `src/bep.d.ts` (generated by `npm run pull`).

### Workflow

1. **Run `npm run pull`** to get the latest types from the BEP. Do this at the start of every session and whenever you know the BEP has been updated on the platform (new effects, resolvers, etc. added). The types in `src/bep.d.ts` must always match the current state of the BEP — never assume they are up to date without pulling first.
2. **Read `src/bep.d.ts`** to understand which IDs exist: `EffectId`, `AutomationId`, `ResolverId`.
3. **Read `src/runtime/index.ts`** to see which handlers are already registered.
4. **Implement missing handlers** — add a file in the appropriate sub-folder and register it in `index.ts`.
5. **Read `src/lenses/mappings.ts`** — it lists every remote data source with its UUID and resolver. Fill in the lens and adapter entries for each one.
6. **Verify** it compiles: `npx tsc --noEmit`.
7. **Build the runtime:** `npm run build:runtime` — inspect `dist/runtime.cjs` if needed.
8. **Deploy the runtime:** `npm run deploy:runtime` (runs the build automatically).
9. **If you added or changed lenses or mappings:** `npm run deploy:lenses`.

### Handler types

#### Effects
Triggered from workflow edges. Receive the active instance and the payload defined in the BEP. Fire-and-forget: the engine does not wait for or react to the result, so effects should be used for actions where staleness on failure is acceptable (notifications, logging), never for state that must stay in sync with an external system (see Automations below for that case).

```typescript
// src/runtime/effects/notify-reviewer.ts
import type { WorkflowInstance } from '@dotbep/core'
import type { BepEffects } from '../../bep.js'

export async function notifyReviewer(
  instance: WorkflowInstance,
  payload:  Parameters<BepEffects['notify-reviewer']>[0],
): Promise<void> {
  // this.env is not available here — pass env as a parameter if needed,
  // or keep the logic in index.ts as an inline arrow function.
}
```

Register in `src/runtime/index.ts`:
```typescript
this.effect('notify-reviewer', notifyReviewer)
```

> If a handler needs `this.env`, define it as an inline arrow function in `index.ts` instead of a separate file.

#### Automations
Triggered on automation nodes — system-executed steps in a workflow. Must return `{ success: true, eventId: string, ...output }` on success, matching a valid outgoing transition of the node, or `{ success: false, error?: string }` on failure. Do not throw for expected failure paths — return `{ success: false, error }` instead so the engine records the attempt; a thrown exception is also caught and recorded as a failure, but returning it explicitly is preferred when the failure is anticipated. The engine tracks failed attempts natively (queryable as `automationPending` on the instance's status) — a following decision node is only needed if the automation's own output carries a genuine business branching need, not to check success or failure.

```typescript
// src/runtime/automations/auto-approve.ts
import type { WorkflowInstance } from '@dotbep/core'
import type { BepAutomations } from '../../bep.js'

export async function autoApprove(
  instance: WorkflowInstance,
): Promise<ReturnType<BepAutomations['auto-approve']>> {
  try {
    const approved = await checkSomeCondition(instance.trackedAsset.id)
    return { success: true, eventId: approved ? 'approved' : 'rejected' }
  } catch (e) {
    return { success: false, error: (e as Error).message }
  }
}
```

#### Resolvers
Fetch external data for the BEP's Live Resources. Take no arguments, return any JSON-serializable value.

```typescript
// src/runtime/resolvers/fetch-issues.ts
export async function fetchIssues(): Promise<{ id: string; status: string }[]> {
  const res = await fetch('https://api.example.com/issues')
  return res.json()
}
```

> Resolvers that need API keys should be defined as inline arrow functions in `index.ts` so they can access `this.env`.

### Environment variables

Handlers that need secrets must be defined as inline arrow functions in `src/runtime/index.ts` to access `this.env`:

```typescript
this.resolver('fetch-issues', async () => {
  const res = await fetch('https://api.example.com/issues', {
    headers: { Authorization: `Bearer ${this.env.API_KEY}` },
  })
  return res.json()
})
```

In production, runtime environment variables are configured on the dotbep platform — not in `.env`, which is otherwise only used by the pull/deploy scripts. The one exception is `npm run dev`: the local preview harness runs resolvers for real (to fetch live data for the lens you're previewing), so it reads the same `this.env` values from your local `.env` — fill in your own test credentials there for any key `npm run pull` adds to `.env.example`.

### Custom lenses

Lenses are browser-side LitElement Web Components. The `hello-lens` component is included as a working example.

To add a new lens:

1. Create `src/lenses/my-lens.ts` with a `@customElement('my-lens')` class.
2. Give it a `@property({ type: Array }) data: YourType[] = []` — the platform passes resolver data through `data`.
3. Define its Zod schema in `src/lenses/schemas.ts` if needed.
4. Export it from `src/lenses/index.ts`.
5. Run `npm run deploy:lenses` to build the bundle.

The tag name (e.g. `my-lens`) is the lens ID used when mapping resolvers to lenses on the platform.

#### Adapters

Adapters are pure synchronous functions that transform resolver output into the format a specific lens expects. They live in `src/lenses/adapters/` and are exported from `src/lenses/index.ts` — they run in the browser alongside the lens components, never on the server.

```typescript
// src/lenses/adapters/issues-to-pie.ts
export function issuesToPie(
  rows: Array<{ status: string }>,
): { label: string; value: number }[] {
  const counts: Record<string, number> = {}
  for (const row of rows) counts[row.status] = (counts[row.status] ?? 0) + 1
  return Object.entries(counts).map(([label, value]) => ({ label, value }))
}
```

Export from `src/lenses/index.ts`:
```typescript
export * from './adapters/issues-to-pie.js'
```

One resolver can have multiple adapters (e.g. one for a table lens, another for a pie chart). The mapping between resolver → adapter → lens is declared in the lenses manifest.

#### Built-in lenses (`dotbep:*`)

The platform provides these out of the box — no custom code needed. Read `src/lenses/built-in.ts` (downloaded by `npm run pull`) for the exact Zod schemas each lens expects. Use those schemas as the authoritative spec when writing adapters.

#### Custom lens design

Before writing any custom lens component, read `DESIGN.md`. It contains the platform design system — tokens, visual patterns, and lens conventions — that all custom components must follow.

`DESIGN.md` is **not** downloaded by `npm run pull`, and you should not write it to disk yourself either — not every session touches lenses, and a runtime may define its own design system instead of the platform's. Resolve it yourself, in this order:

1. If `DESIGN.md` already exists at the repo root, read it — that means the runtime owner committed one intentionally to override the platform's, and it always wins.
2. Otherwise, fetch `https://app.dotbep.com/_kit/DESIGN.md` and read it straight from the response, in memory, for this session only. Do not save it as a file — a saved copy would be indistinguishable from an intentional override on the next session, permanently blocking the platform's design system from ever being picked up again.

### Important rules

- **Do not edit `src/bep.d.ts`** — it is overwritten by `npm run pull`.
- **Do not deploy:runtime without compiling** — verify with `npx tsc --noEmit` first.
- **Delete the `hello-*` examples** once the real handlers are in place.
- **Prefer direct REST API calls over third-party SDKs.** The runtime is bundled into a single file — every SDK you add inflates the bundle size. Use `fetch` against the service's REST API instead. For example, call the GitHub REST API directly rather than installing `@octokit/rest`, or the Notion API directly rather than `@notionhq/client`. Only add a package to `dependencies` when there is no reasonable REST alternative (e.g. a library that handles binary encoding or crypto).
- **Prefer native platform APIs over utility libraries.** Before adding a dependency, ask: can this be done with what the runtime/browser already provides? If yes, write it inline. A charting library is justified when the visualization is genuinely complex; it is not justified to format a date or sum an array. Every package added inflates the bundle — the bar for inclusion should be "I couldn't reasonably write this myself", not "this library does it".
- Keep handlers focused. Complex logic goes in helper functions in the same file.
